# Фреймворк по тестированию мобильных приложений

## Что реализовано

- Программный запуск Appium Server: `src/main/java/mobile/AppiumServerLauncher.java`.
- Создание и управление сессией: `src/main/java/mobile/DriverManager.java`.
- Сессия параметризована через TestNG-параметры `appPackage` и `appActivity` или подтягивается по тествому классу если запуск через Run теста.
- Использован паттерн Page Object Model (POM).
- Page Objects: `src/main/java/mobile/pageObjects/`.
- Тесты: `src/test/java/mobile.tests/`.
- Подключены Allure Steps (`@Step`) для методов page objects.

## Предусловия

- На эмулятор уже установлены тестируемые приложения:
  - `com.vk.vkvideo`
  - `com.ilyin.alchemy`
- Доступен Android-эмулятор/устройство и ADB.
- Appium поднимается автоматически из кода.


## Allure отчет

- Результаты: `target/allure-results`
- Генерация/просмотр:
  - `mvn allure:serve`

> Важно: для `allure:serve` должен быть корректно настроен `JAVA_HOME` (путь к корню установленного JDK).

## Notes

### Alchemy

Тест по Alchemy уязвим из-за рекламы и ее нестабильного/непредсказуемого поведения:

- Часть кейсов обработана, но в реальном проекте желательно иметь сборку без рекламы (или с замоканной рекламой).
- Иначе тест либо не подлежит надежной автоматизации, либо требует полного набора формальных правил/состояний для скипа рекламы.
- В приложении почти нет стабильных `id`, что затрудняет построение надежных локаторов.

### VK Video

В `VkvideoTest` используется явное ожидание (`sleep`/`waitExactTime`), потому что:

- Нужно проверить именно факт воспроизведения во времени.
- Таймер прогресса в accessibility tree отображается только когда видео стоит на паузе.
